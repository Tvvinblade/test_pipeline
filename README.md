# Data Pipeline: Airflow + ClickHouse + Kafka + Superset

Проект демонстрирует **end-to-end** пайплайн обработки данных от JSON-источника до аналитических дашбордов в Superset, используя Kafka для стриминга и ClickHouse как основное хранилище.

## Архитектура пайплайна

Поток данных в системе выглядит следующим образом:

```text
Источник данных (JSON)
        ↓
    Kafka Broker
        ↓
Raw Layer (ClickHouse MergeTree)
        ↓
Airflow DAGs (трансформация, обогащение)
        ↓
Analytics Layer (ClickHouse MergeTree)
        ↓
Superset (дашборды и визуализация)



Компоненты
Источник данных (JSON)
Внешние сервисы или генераторы событий формируют сообщения в формате JSON (например, события приложения или логирование действий пользователей) и отправляют их продюсером в Kafka.

Kafka Broker
Kafka выступает шиной данных и буфером между продюсерами и потребителями: обеспечивает надёжную доставку, хранение оффсетов и возможность реплея сообщений для повторной обработки.

Raw Layer (ClickHouse, MergeTree)
Отдельный потребитель (или таблица Kafka Engine + materialized view) читает сообщения из Kafka и сохраняет их в «сырые» таблицы ClickHouse на движке MergeTree.
На этом уровне данные хранятся максимально близко к исходному JSON без сложных трансформаций.

Airflow DAGs (трансформация и обогащение)
Apache Airflow оркестрирует DAG’и, которые:

периодически читают данные из raw-таблиц;

выполняют очистку, нормализацию и обогащение (join с референсными данными, денормализация, агрегирование);

записывают подготовленные данные в аналитический слой ClickHouse.

Analytics Layer (ClickHouse, MergeTree)
Аналитические таблицы содержат предобработанные и агрегированные данные, оптимизированные под запросы BI: настроено партиционирование (обычно по дате), сортировочные ключи и индексы под основные сценарии чтения.

Superset (дашборды и визуализация)
Apache Superset подключается к ClickHouse как к источнику данных и использует аналитический слой для построения чартов и дашбордов.
Аналитики и бизнес‑пользователи работают только с этим слоем, не обращаясь напрямую к raw-таблицам.

Основной сценарий использования
Запустить инфраструктуру (Kafka, ClickHouse, Airflow, Superset) через Docker / docker-compose.

Отправить тестовые JSON-события в Kafka-топик (скриптом-продюсером или утилитами Kafka).

Убедиться, что события попадают в raw-таблицы ClickHouse.

Запустить/дождаться выполнения DAG’ов в Airflow и проверить заполнение аналитических таблиц.

Настроить в Superset dataset’ы на аналитический слой и открыть готовые дашборды.
